## Summary

Implemented the canonical, server-side feature/add-on entitlement read used by
`requireFeatureAccess`, wired it into `functions/lib/auth.ts`'s `entitlementCheck`,
and unit-tested it. Following the **task-23 precedent**, applying the guard to
**live handlers is deferred** (that is task 24.2) because the entitlement shadow
caches are not yet broadly populated by the SSO entitlement sync/backfill — wrapping
real endpoints now would 403 real users.

## What was implemented

- **New module:** `functions/lib/entitlements.ts`
  - `hasAnyFeature(supabase, userId, keys)` — design's contract; grants when ANY
    key resolves; **fails closed** on missing `userId`, empty/invalid `keys`, DB
    errors, or unexpected throws.
  - `hasFeatureEntitlement(supabase, userId, featureKey)` — composes:
    1. subscription plan + freemium baseline via the existing
       `checkServerFeatureAccess` (`functions/shared/lib/server-feature-gating.ts`,
       reads `subscription_cache` + `plans_cache`, self-healing), then
    2. purchased add-ons via `hasActiveAddonEntitlement`.
  - `hasActiveAddonEntitlement(supabase, userId, featureKey)` — reads
    `user_entitlements` (app-DB shadow of SSO `addon_purchases`/`bundle_purchases`),
    granting only when `status ∈ {active, grace_period}`.
- **Wiring:** `functions/lib/auth.ts` → `entitlementCheck(context, keys)` now calls
  `hasAnyFeature(getServiceClient(env), getContextUser(context).id, keys)`,
  replacing the prior fail-closed `return false` scaffold. `requireFeatureAccess`
  remains `requireFeature(featureKey, entitlementCheck, handler)`.
- **Tests:** `functions/lib/__tests__/entitlements.test.ts` (12 unit tests, all passing) —
  plan-included, freemium baseline, add-on fallback, deny paths, and fail-closed
  cases (missing user id, empty/invalid keys, DB/missing-table error).

## Tables / columns read (canonical state SSO-side)

| Table | Columns read | Filter | Purpose |
|---|---|---|---|
| `subscription_cache` | `id, status, plan_code, features, synced_at` | `user_id =`, `status in (active, grace_period)` | active plan + plan `features[]` (via `checkServerFeatureAccess`) |
| `plans_cache` | (via `checkServerFeatureAccess` plan lookups) | — | plan feature reference |
| `user_entitlements` | `id` | `user_id =`, `feature_key =`, `status in (active, grace_period)` | purchased add-on entitlement |

- **Fail-closed**: any ambiguity → DENY (CC-2: a guard never silently opens).
- Reads **only** the server-side shadow (never frontend-supplied flags).
- Parameterized query builders only (no string interpolation).
- No long-lived cache beyond the shadow tables (preserves CC-1 staleness bound).

## DB findings (local app DB, `127.0.0.1:54322`)

Read-only inspection on 2026-06-07:

- `subscription_cache`: **1 row** — single user on `freemium` (`status=active`,
  `features=[dashboard_access, profile_creation, marketplace_access, view_pricing,
  opportunities_access, courses_listing_access]`).
- `plans_cache`: **4 rows**.
- `user_entitlements`: **0 rows** (no purchased add-ons present).

So today entitlement enforcement would grant only freemium baseline features to one
user and **deny every paid/add-on feature for all users**. The caches are not yet
populated by a broad SSO entitlement sync/backfill.

## Deferral decision (task-23 precedent, user-approved)

- **24.1 (this task):** implement + wire + unit-test the canonical `entitlementCheck`
  read — **done**.
- **24.2 (deferred):** wrapping live feature-gated handlers with
  `requireFeatureAccess(featureKey, …)` is **NOT** done here. Live wrapping waits
  until the SSO entitlement sync/backfill populates `subscription_cache`/
  `plans_cache`/`user_entitlements`, so real users are not 403'd.
- No live-handler 403 behavior was introduced. `requireFeatureAccess` is applied to
  zero handlers, identical to before this task.
