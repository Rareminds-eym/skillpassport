# SkillPassport To LTE SSO Implementation Plan

## Goal

Implement the SkillPassport side of the `SkillPassport -> LTE -> Dashboard` flow, where SkillPassport never sends access or refresh tokens through the browser URL and only asks SSO for a short-lived one-time LTE authorization code.

## Final Flow

1. User is already authenticated in SkillPassport.
2. User clicks `Go to LTE`.
3. SkillPassport frontend calls SkillPassport backend.
4. SkillPassport backend verifies the current SkillPassport session.
5. SkillPassport backend asks SSO to generate an LTE authorization code.
6. SSO returns `code`, `state`, `redirectUrl`, and expiry metadata.
7. SkillPassport frontend stores `state` temporarily for navigation validation.
8. Browser navigates to LTE `/auth/callback?code=...&state=...`.
9. LTE owns the rest of the exchange and dashboard session.

## SkillPassport Responsibilities

Authenticate the current SkillPassport user before generating an LTE handoff.

Call SSO through the existing `SSO_SERVICE` binding.

Pass only `code` and `state` to LTE through the URL.

Never expose SkillPassport access token, SkillPassport refresh token, LTE access token, or LTE refresh token in URL, localStorage, or sessionStorage.

Use `SameSite=Lax` host-only SkillPassport refresh cookie behavior for normal authenticated frontend navigation.

## Data Ownership

SkillPassport access token - Stored only in SkillPassport frontend memory.

SkillPassport refresh token - Stored only in SkillPassport HttpOnly host-only cookie.

LTE authorization code - Created by SSO and stored in SSO Durable Object.

LTE state - Returned by SSO and temporarily stored by SkillPassport frontend only to complete navigation.

LTE tokens - Not created, stored, or read by SkillPassport.

## Backend Endpoint Contract

`POST /api/auth/generate-lte-code` - Requires authenticated SkillPassport session and returns only an LTE redirect URL plus code expiry metadata.

Response body should include `redirectUrl`, `codeExpiresAt`, and optionally `stateExpiresAt`.

Response body must not include access tokens, refresh tokens, password data, or raw user profile records.

## SSO RPC Contract Used By SkillPassport

`generateAuthorizationCode` - Receives SkillPassport access token, target app `lte`, redirect URI, IP, and user agent.

SSO verifies the caller, checks LTE product entitlement, creates one-time code/state, stores hashes in Durable Object, and returns raw `code` and `state`.

## SkillPassport Frontend Behavior

Call `generate-lte-code` only after user action.

Store returned `state` only in `sessionStorage` for short-lived navigation validation.

Navigate with `window.location.href` to LTE redirect URL.

Show a normal error message if SSO denies LTE access.

## Security Rules

Only authenticated SkillPassport users can request LTE code generation.

Only `targetApp = lte` is allowed for this flow.

Redirect URI must be server-selected or validated against allowlist.

Authorization code TTL must remain short, around 60 seconds.

State must be generated by SSO and validated during LTE exchange.

No token may be added to URL query parameters.

No refresh token may be returned in JSON.

## Implementation Order

1. Confirm SkillPassport `SSO_SERVICE` binding to `sso-api`.
2. Add typed `generateAuthorizationCode` wrapper in `functions/lib/sso-client.ts`.
3. Add `functions/api/auth/generate-lte-code.ts`.
4. Add frontend API helper for generate LTE code.
5. Add `navigateToLTE` frontend helper.
6. Wire `Go to LTE` action in learner dashboard header or profile menu.
7. Add user-facing unauthorized/error behavior when LTE entitlement is missing.
8. Verify browser URL contains only `code` and `state`.
9. Verify SkillPassport tokens never leave SkillPassport storage rules.

## Verification Checklist

SkillPassport authenticated user can click `Go to LTE`.

Unauthenticated user cannot generate LTE code.

User without LTE entitlement receives a clear denied response.

Generated LTE URL contains `code` and `state` only.

SkillPassport access token is not in URL.

SkillPassport refresh token is not in URL or JSON.

State is stored only temporarily for navigation.

SSO receives correct `targetApp = lte`.

LTE callback route opens after navigation.
